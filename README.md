# 🚗 中古車マルチサイトリアルタイム監視システム

## 📌 概要

主要な中古車販売サイト、フリマアプリ、およびSNSから条件に合致する車両情報を一括監視し、Discordへ即時通知する統合自動化ツールです。
アプリ通知に依存せずページ構造を直接解析することで、出品直後の車両情報を早期に取得することを目的としています。

## 🌟 開発の背景と目的

中古車、特に希少車や格安車両等条件の厳しい車両の購入において、「情報の鮮度」はとても重要なものになります。
本プロジェクトは、大手サイトのアプリ通知に依存しない監視を行い、かつ複数のプラットフォーム（信頼性の大手・コスパのフリマ・最安のSNS）を横断して「市場の最速」をキャッチするために開発されました。

- **中古車大手（カーセンサー/グーネット）**: 信頼性の高いショップ情報を最速で捕捉
- **フリマ・オークション（ヤフオク/メルカリ/ジモティー）**: 個人売買による割安・訳あり物件を網羅
- **SNS（X/Twitter）**: 手数料のない個人間取引や、未公開の掘り出し情報を捕捉

元々は自分用ツールとして開発しましたが、将来的なサービス提供も見据えた設計にリファクタリングしています。

## 🎯 このツールで実現できること

- ✅ カーセンサー・グーネットの新着車両をアプリ通知に依存しない監視
- ✅ ヤフオク・メルカリ・ジモティーの3サイトを一括監視
- ✅ X（旧Twitter）のキーワード検索で個人売買も逃さない
- ✅ 5〜10分間隔の自動実行でほぼリアルタイムにDiscord通知
- ✅ GitHub Actions で完全自動化（サーバー不要・クラウド料金不要）

## 🛡️ 堅牢な設計

- **重複通知の完全排除**：履歴ファイルによるID管理
- **サイト構造変化に強い**：柔軟な要素特定ロジック（lambda、正規表現）
- **エラー時の自動リトライ**：通信失敗時も安定動作
- **1サイトが失敗しても全体は停止しない**：エラー分離設計
- **レート制限対策**：適切な待機時間（2〜5秒）による負荷分散

## 🛠 技術的特徴とこだわり

本ツールは単なるスクレイピングツールではなく、実用性・保守性・可用性を重視した設計を行っています。

### 1. ハイブリッド・スクレイピング・エンジン

サイトの特性に合わせて最適なライブラリを使い分けることで、速度と確実性を両立しています。

#### 技術選定マトリックス

| カテゴリ         | 対象サイト                | 取得手法            | 選定理由                             |
| :--------------- | :------------------------ | :------------------ | :----------------------------------- |
| **中古車大手**   | カーセンサー / グーネット | Requests + BS4      | 構造がシンプルで高速レスポンスが可能 |
| **オークション** | ヤフオク                  | Requests + BS4      | 安定したHTML構造からの高速取得       |
| **フリマ**       | メルカリ / ジモティー     | Playwright + BS4    | 動的レンダリングに対応               |
| **SNS**          | X (Twitter)               | Playwright + Nitter | API制限を回避し、持続的な監視を実現  |

**技術的なメリット**：

- JavaScript実行後のDOMを確実に取得
- 実ブラウザ環境での取得により安定性向上（適切なUser-Agent設定・viewport設定）
- タイムアウトによる無限待機の防止

### 2. サイト構造変化に強い柔軟な抽出ロジック

#### **lambda による部分一致検索**

- サイトのマイナーアップデートで検索するワードが多少変わっても動作継続
- 複数のクラス名が付与されているケースにも対応

#### **親要素への逆引き**

- 価格・タイトル・リンクが同じ階層にない複雑な構造に対応
- 商品枠全体を取得してから、必要な情報を抽出

### 3. 重複排除の多層防御

- レベル1：ページ内重複排除（set管理）
- レベル2：履歴ファイル管理（永続化）
- レベル3：通知成功時のみ履歴保存

再実行しても同じ通知が飛ばない設計。

**技術的なメリット**：

- 履歴ファイルによる永続的な管理
- 通知送信成功時のみ履歴に保存（送信失敗時は再通知）

### 4. 堅牢なエラーハンドリングと可用性

- **Nitterインスタンス・ローテーション**
  - **技術的なメリット**：
    - X（旧Twitter）公式APIは有料 → 本ツールは完全無料
    - 特定サーバーのダウンに備えたフェイルオーバー機能
    - レート制限なし・アカウント不要
- **エラー分離設計**
  - 1サイトが失敗しても全体は停止しない設計により、安定した継続稼働を実現。

### 5. 行儀の良いスクレイピング（Anti-Bot Policy）

本ツールは以下のマナーを守って設計されています：

- ✅ ランダムな待機時間：`time.sleep(random.uniform(2, 5))`
- ✅ 適切な User-Agent 設定：実際のブラウザとして認識される
- ✅ サイト間インターバル：連続アクセスを避ける（3秒間隔）
- ✅ Discord APIのレート制限対策：通知送信後に1秒待機

## 設計思想

- 各サイトを独立したモジュールに分離（保守性・拡張性）
- 共通処理を `main.py` に集約（通知・履歴管理）
- エラー発生時も他サイトの処理は継続（フォールトトレラント設計）
- 新規サイト追加は `fetch_listings()` を実装し、`SITES_CONFIG` に登録するだけ

## 📦 セットアップ手順

1. **リポジトリのクローン**

   ```bash
   git clone [https://github.com/YOUR_USERNAME/your-repo-name.git](https://github.com/YOUR_USERNAME/your-repo-name.git)
   cd your-repo-name

   ```

2. **依存ライブラリのインストール**

   ```bash
   pip install -r requirements.txt
   playwright install

   ```

3. **Discord Webhook URLの設定**
   - Discordサーバーで「設定 → 連携サービス → ウェブフック」を開く
   - 「新しいウェブフック」を作成し、URLをコピー
   - プロジェクトルートに `.env` ファイルを作成：`DISCORD_WEBHOOK_URL = あなたのWebhookURL`

4. **検索条件のカスタマイズ**
   `main.py` の `SITES_CONFIG` を編集：
   ```python
   SITES_CONFIG = [
       {
           "name": "カーセンサー",
           "url": "[https://www.carsensor.net/usedcar/search.php](https://www.carsensor.net/usedcar/search.php)?...",  # ←検索条件URL
           "history": "history_carsensor.txt",
           "fetcher": fetch_carsensor
       },
       # ...
   ]
   ```

**検索条件URLの取得方法：**

- 各サイトで条件を指定して検索
- 検索結果ページのURLをコピー
- `main.py` の該当箇所に貼り付け
- **注意**：グーネットは検索条件の保存方法が特殊なため、正しくURLを取得できない場合があります。その際は、サイト内の「お気に入り保存」機能を活用してください。
